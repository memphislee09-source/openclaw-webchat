## 2026-03-15
- 前端脚本拼接错误 / 原因：多次局部编辑后 `public/app.js` 出现重复片段，触发 JS 语法错误 / 修复：重写 `public/app.js`，恢复单一入口与完整函数结构 / 预防措施：前端大改优先整文件覆盖，改后立即 `node --check public/app.js`
- 发图轮次过早超时 / 原因：等待 assistant 最终回复窗口仅 25s，长耗时图片生成场景会先返回占位文案 / 修复：增加 `OPENCLAW_WEBCHAT_ASSISTANT_WAIT_TIMEOUT_MS`，默认提升到 120s / 预防措施：把长耗时媒体生成纳入自测与专项回归
- gateway 临时断连 / 原因：`openclaw gateway call` 遇到 normal closure / connect failed 时会直接失败 / 修复：服务端增加 gateway 调用重试 / 预防措施：保留重试，并在验收时覆盖多 agent / 多轮发送场景
- `/new` 后旧异步结果串入新会话 / 原因：上游 session key 固定为 `agent:<id>:openclaw-webchat:main`，reset 后晚到 completion 仍会落入当前 key；同时旧代次回包没有在接入层被丢弃 / 修复：`/new` 时轮换 upstream session key，并在服务端按代次忽略旧 key 的迟到回包 / 预防措施：自测新增断言，要求 `/new` 后 `upstreamSessionKey` 必须变化，避免再次把 reset 做成“同 key 清空”

## 2026-03-16

- 本地 slash 命令的非法参数被当成接口错误 / 原因：`runSlashCommand(...)` 在 `/think`、`/fast`、`/verbose` 参数无效时直接抛错，前端只能收到 HTTP 500，而不是原生 WebUI 风格的本地提示 / 修复：服务端改为本地生成 assistant 提示消息，并补齐查询/设置分支与 `sessions.list`、`models.list`、`sessions.patch`、`sessions.compact` 语义 / 预防措施：把 slash 参数校验视为正常用户路径，优先返回本地提示消息；自测覆盖 `/model`、`/think`、`/new`
- 跨 agent 会话串 processing 状态与最新回复 / 原因：前端把 `sending` 和 `messages` 作为全局当前视图状态使用，异步发送回包、slash 回包、历史加载都没有绑定发起时的 `sessionKey`；切换会话后，旧请求结果会写入当前打开的会话 / 修复：改为按 `sessionKey` 维护发送中状态，并为发送、slash、历史加载、会话打开都增加上下文校验，只在请求仍属于当前会话时才更新 UI / 预防措施：前端所有异步 UI 写入都必须携带会话上下文；手工回归必须覆盖“处理中切换 agent”的竞态场景
- 图文混排时图片不显示或全部堆到消息末尾 / 原因：服务端 `parseTextIntoBlocks(...)` 会把正文里提及 ``MEDIA:`` 的说明文字误判为真实媒体指令，同时不识别 `![alt](url)` 的 Markdown 图片；前端 `renderMessages()` 又把全部文本和全部媒体分开渲染，破坏原始顺序 / 修复：服务端仅在右侧值确实像 URL/路径时才接受 `MEDIA:` / `mediaUrl:`，并补充 Markdown 图片提取；前端改为按 block 原始顺序逐个渲染文本与媒体 / 预防措施：今后涉及消息解析的改动必须同时校验“解析正确性”和“渲染顺序”，人工回归增加“正文中交错多图”的场景
- 移动端历史加载连续试探但未根治 / 原因：前端历史分页当前仍绑定在消息区内层滚动容器的 `scrollTop` 上；在手机浏览器里，这个前提并不稳定，导致基于 touch、顶部按钮、自动补页的临时兜底都不能证明真正根因 / 修复：已回收会影响桌面端的临时历史兜底改动，保留基线交互；当前仅保留静态资源 no-store 和版本戳以降低缓存干扰 / 预防措施：下一轮应直接从滚动架构入手，先验证“移动端实际滚动容器是谁”，再决定是否改为单滚动容器或顶部哨兵触发，避免继续堆叠表层补丁
- 移动端首屏打开体感偏慢 / 原因：虽然并未一次性下发全部历史，但首屏打开固定返回 `30` 条消息，在手机上首次同步渲染 Markdown / 媒体卡片的成本偏高 / 修复：在测试分支 `codex/mobile-history-test` 中把 `/open` 的初始历史页下调为 `15` 条，后续历史分页仍保留 `30` 条；同时小屏隐藏顶部“刷新”和副标题，减少头部占位 / 预防措施：移动端性能优化优先拆分“首屏载荷”和“后续分页”，不要把分页速度与首屏打开成本混为一谈；实验性参数先放测试分支观察，再决定是否回合主线

## 2026-03-17

- agent 头像设置后过一段时间失效 / 原因：设置页保存头像时把 `/api/openclaw-webchat/media?token=...` 这种 15 分钟有效的签名 URL 直接持久化到了 profile；token 过期后头像展示随之失效 / 修复：前端保存时改为提交上传返回的稳定 `source` 路径，服务端在读取 profile 时再动态签发新的媒体 URL；同时兼容把历史上已持久化的旧 token 反解回文件路径 / 预防措施：凡是要长期持久化的媒体引用，一律存稳定源路径，不存短时签名 URL；自测已新增头像持久化回归断言

## 2026-03-18
- 媒体 token 默认 secret 固定且本地媒体白名单过宽 / 原因：服务端在未配置环境变量时回退到固定字符串 `openclaw-webchat-local-secret`，同时媒体路径白名单接受整个 `$HOME`，导致拿到源码或默认配置的人可以伪造 token 读取用户目录内文件 / 修复：改为在 data 目录持久化随机 media secret（优先读环境变量，其次读写 `.media-secret`），并将本地媒体白名单收紧到上传目录与 `~/.openclaw`，同时补充真实路径校验 / 预防措施：签名 secret 不能依赖公开默认值；本地媒体代理白名单必须按目录最小化，并使用真实路径防止软链逃逸
- 文件类媒体链接未限制 scheme / 原因：agent 返回的 `url/href` 会被原样渲染到前端文件卡片，缺少 `http/https` 白名单与非法 scheme 拦截 / 修复：服务端输出阶段统一校验远端媒体地址，只允许 `http/https` 和受控的本地签名媒体地址；非法 scheme 现在回退为不可访问卡片 / 预防措施：所有可点击 URL 在进入前端前都应统一做协议校验
- assistant 超时占位消息污染正式历史 / 原因：等待超时时会先把 `（处理中，稍后自动补回）` 作为 assistant 正式消息写入 JSONL，迟到真实回复补回后就会形成重复轮次 / 修复：将该占位改为只返回给当前前端会话的临时 assistant 消息，不再写入正式历史；迟到真实回包只落真实回复 / 预防措施：所有“处理中”占位都应与正式历史写入解耦
- 音频转写同步阻塞请求线程 / 原因：上传接口里使用 `spawnSync` 调 Whisper，长音频会阻塞整个 Node 事件循环，影响聊天、历史和 healthz / 修复：转写链路改为异步 `execFile` 子进程调用，保留超时与错误兜底，但不再阻塞整个 Web 请求线程 / 预防措施：CPU 或外部命令型任务不能在 Web 请求链路里用同步 API
- 自测脚本缺少统一请求超时 / 原因：`scripts/selftest.mjs` 直接使用裸 `fetch`，异常环境下会长时间挂起，不利于快速定位问题 / 修复：补统一 `AbortSignal.timeout(...)` 控制，并支持 `OPENCLAW_WEBCHAT_SELFTEST_REQUEST_TIMEOUT_MS` 覆盖 / 预防措施：所有回归脚本网络请求都应带超时上限
- 聊天页自动跟随滚动存在回归风险 / 原因：消息区当前主要依赖多处直接 `scrollMessagesToBottom()` 调用，缺少统一“用户是否仍希望贴底”的状态管理，处理中/媒体加载/轮询刷新叠加时容易出现跟随不稳定 / 修复：新增统一 `autoScrollPinned` 状态，滚动事件实时维护贴底偏好；发送、slash、本地处理中和媒体加载统一经 `maybeScrollMessagesToBottom(...)` 决策，避免多处分散硬滚到底 / 预防措施：消息区自动滚动应统一收口，避免多处分散调用互相覆盖
- 旧头像 token 在新 secret 策略下无法反解 / 原因：更早期 profile 中仍可能持久化旧版签名头像 URL；服务端改为随机 secret 后，这些历史 token 无法被当前 `decodeMediaToken(...)` 反解回稳定文件路径 / 修复：在头像迁移链路中对历史默认 secret 保留受限兼容，只用于 profile 里的旧 token 反解为稳定路径；媒体访问接口仍只接受当前 secret 签发的新 token / 预防措施：涉及签名策略切换时要保留受限迁移通道，确保已持久化配置不会断裂，同时不要恢复旧 token 的通用访问能力
- 左侧 agent 列表整列重绘导致身份信息被频繁刷新 / 原因：当前 `refreshAgents()` 每次轮询都直接 `renderAgentList()` 清空并重建整列 DOM，头像和显示名也会跟随重复刷新；这与“身份信息由设置页主动推送，摘要/状态才轮询刷新”的交互预期不一致 / 修复：左栏改为局部 diff 复用现有卡片 DOM；轮询只更新 presence、最新回复摘要和时间，设置保存时才主动刷新头像与名称 / 预防措施：轮询列表应区分“静态身份信息”和“动态会话状态”，避免全量重绘

## 2026-03-24
- Athena 在 Claw WebChat 中不会把已生成图片正确发回 / 原因：当前隐藏 bootstrap 虽然提到了 `MEDIA:` / `mediaUrl:` fallback，但约束不够明确；`gpt-5.4` 会优先尝试 `message` 工具并使用不存在的 `webchat` channel，最终误判为“当前没有可用的 webchat 附件发送通道” / 修复：将隐藏 bootstrap 收短并明确写死本地文件与直链远程 `http/https` 媒体的正确回传方式，要求优先结构化 media block，fallback 时使用单独一行 `MEDIA:<path-or-url>` / `mediaUrl: <path-or-url>`，并明确禁止 `message` / `webchat` 发送路径；同时提升 `BOOTSTRAP_VERSION` 让现有会话自动补吃新版约定，并在自测中增加本地路径与远程 URL fallback 断言 / 预防措施：渠道级媒体协议提示必须同时满足“足够明确”和“足够短”；一旦发现某类模型倾向走错误工具路径，应优先收紧 bootstrap 文案并补静态回归，避免靠逐个 agent 手工提醒
